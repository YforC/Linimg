# 图片与视频生成工作台设计

**日期：** 2026-03-26

**目标：** 基于 Next.js 构建一个单页前端工作台，用于图片生成与视频生成，采用简约自然风格，并通过服务端 API Route 代理 OpenAI SDK 风格接口。

## 1. 项目背景

当前项目目录基本为空，仅有一个空的 `api.md` 文件，没有现成的 Next.js 工程结构，也没有可复用的前端页面或接口层代码。

因此本次设计默认从零开始搭建一个 Next.js 应用骨架，并围绕以下原则组织页面：

- 单页工作台，而不是两个独立页面
- 页面同时承载图片生成与视频生成能力
- 前端不直接暴露密钥，所有真实调用通过 Next.js 服务端路由完成
- 视觉语言偏自然、克制、留白充足，避免重科技感和夸张装饰

## 2. 产品定位

该页面定位为“折中型创作工作台”：

- 以创作输入为主，突出 prompt 和结果展示
- 保留少量常用高级参数，但默认不过度暴露复杂配置
- 支持用户在同一页面里完成图片和视频两类内容生成

用户体验目标：

- 用户能快速理解页面用途并立即开始创作
- 图片和视频共享创作语境，但参数和结果展示清晰分离
- 交互自然、轻量，不让用户产生进入复杂后台系统的压迫感

## 3. 信息架构

页面采用单页纵向布局，从上到下分为五个区块。

### 3.1 顶部 Hero 区

作用：

- 用最少文字解释这是一个图片与视频创作工作台
- 建立简约自然的视觉基调
- 将用户注意力引导到主输入区域

内容建议：

- 主标题：强调“生成图像与视频”能力
- 副标题：强调通过自然语言描述完成创作
- 简洁说明文案：突出“一个 prompt，可切换两种生成方式”

### 3.2 统一创作输入区

作用：

- 作为整个页面的核心输入入口
- 承接用户的创作意图

内容建议：

- 一个较大的主 prompt 文本输入框
- 若干灵感提示词卡片，帮助用户快速启动
- 辅助输入提示，例如场景、风格、镜头、光线、主体等方向

设计要求：

- 文本框应具有足够视觉权重
- 初次进入页面时，输入区应是页面最显眼的交互点
- 支持清空输入与一键复用历史 prompt

### 3.3 能力切换条

作用：

- 在不离开当前页面的前提下切换生成模式

模式：

- 生成图片
- 生成视频

交互要求：

- 两个模式入口始终可见
- 当前模式高亮显示
- 切换时不清空主 prompt
- 只切换参数面板、主按钮文案和结果展示优先内容

### 3.4 主工作台

工作台采用左右分栏。

左侧：

- 当前模式对应的参数面板
- 参数默认展示常用项
- 高级参数采用折叠展开方式

右侧：

- 当前模式对应的结果展示区
- 初始态、加载态、成功态、失败态统一在此区域切换

图片模式建议参数：

- 图片尺寸
- 输出数量
- 质量或清晰度倾向
- 风格倾向

视频模式建议参数：

- 视频时长
- 输出比例
- 分辨率
- 运动强度或镜头动态倾向

### 3.5 历史结果区

作用：

- 展示本次会话中已生成的图片和视频结果
- 允许用户快速回看和复用 prompt

交互要求：

- 采用时间倒序
- 每个结果卡片显示类型、缩略图或封面、时间信息
- 支持“再次生成”或“复用 prompt”

## 4. 视觉设计

## 4.1 风格方向

整体视觉语言为“简约自然”。

要求：

- 避免赛博、荧光、高对比科技风
- 避免重阴影、过多悬浮层和复杂装饰
- 强调柔和留白、克制色彩、平静气质

### 4.2 色彩建议

建议基于以下方向建立设计变量：

- 背景主色：浅米白、暖灰白
- 辅助底色：雾灰、浅卡其
- 品牌强调色：苔绿、橄榄绿、深森林绿
- 文字主色：深灰褐或深橄榄灰
- 边框与分隔色：低对比暖灰

### 4.3 布局和组件气质

- 卡片使用较大圆角
- 边框细且淡
- 阴影极轻，仅用于层级区分
- 背景可使用非常轻的径向渐变或柔和色块，不使用厚重纹理
- 标题层级清晰，但不过度放大
- 主按钮具有明确存在感，次按钮保持低干扰

## 5. 关键交互设计

### 5.1 默认状态

- 页面默认进入“生成图片”模式
- 输入区为空时，结果区展示空状态说明
- 历史区为空时显示引导文案

### 5.2 生成流程

1. 用户输入 prompt
2. 用户选择生成图片或生成视频
3. 用户调整必要参数
4. 用户点击主按钮发起生成
5. 页面进入加载态
6. 成功后在结果区显示内容，并将结果加入历史区

### 5.3 状态设计

结果区统一支持以下四类状态：

- 初始态：提示输入内容开始创作
- 生成中：展示骨架屏、进度提示或处理中说明
- 成功态：展示图片或视频结果
- 失败态：展示简洁错误文案和重试入口

### 5.4 历史复用

历史记录需要支持以下能力：

- 一键回填 prompt
- 基于历史结果再次发起生成
- 允许用户快速在图像与视频结果间切换浏览

## 6. 技术架构

项目采用 Next.js App Router。

架构划分如下：

### 6.1 前端页面层

负责：

- 页面布局
- 交互状态管理
- 表单输入与参数控制
- 结果展示
- 历史记录展示

不负责：

- 直接保存 API Key
- 直接在浏览器中调用底层 SDK

### 6.2 服务端 API Route 层

建议提供两个服务端路由：

- `/api/generate-image`
- `/api/generate-video`

职责：

- 从环境变量读取 API Key
- 接收前端提交的 prompt 与参数
- 使用 OpenAI SDK 风格接口发起请求
- 将底层返回统一整理为前端可消费的数据结构
- 处理异常并返回规范化错误

### 6.3 接口边界原则

前端只依赖统一后的响应结构，不感知底层 SDK 的细节。这样后续即使模型名、参数格式或响应结构变化，也只需调整服务端路由实现。

## 7. 数据流设计

页面建议维护以下状态：

- 当前 prompt
- 当前模式（图片或视频）
- 当前参数集
- 当前请求状态
- 当前结果
- 本次会话历史记录

数据流如下：

1. 用户在前端填写 prompt 与参数
2. 前端根据当前模式调用对应 API Route
3. API Route 使用服务端密钥调用 OpenAI SDK 风格接口
4. API Route 返回统一格式结果
5. 前端更新结果展示与历史记录

## 8. 错误处理

需要覆盖以下错误场景：

### 8.1 输入校验错误

例如 prompt 为空或明显不合法。

策略：

- 在前端直接拦截
- 就近展示提示文案
- 不发起请求

### 8.2 接口调用失败

例如服务端 4xx、5xx，或第三方接口调用失败。

策略：

- API Route 将底层错误转换为统一结构
- 前端展示可读错误信息
- 提供重试按钮

### 8.3 返回结果为空或格式异常

策略：

- 由服务端路由做统一兜底
- 前端只消费明确结构，不直接解析不稳定字段

### 8.4 重复提交

策略：

- 请求过程中主按钮进入 loading 状态
- 禁止重复点击
- 保持当前表单内容可见

## 9. 测试范围

建议至少覆盖以下测试：

### 9.1 页面渲染测试

- 首页是否正常渲染
- 默认是否进入图片模式

### 9.2 模式切换测试

- 从图片切换到视频时，参数区内容是否变化
- prompt 是否被保留

### 9.3 表单提交测试

- 图片模式是否调用图片生成路由
- 视频模式是否调用视频生成路由

### 9.4 服务端路由测试

- 成功返回时是否输出统一结构
- 失败返回时是否输出统一错误格式

### 9.5 状态展示测试

- 空状态是否显示
- 加载态是否显示
- 失败态是否显示
- 成功态是否正确展示结果

## 10. 文件结构建议

如果后续进入实现，建议采用职责清晰的结构，避免将所有页面逻辑堆积在一个文件中。

建议至少包含：

- 页面入口文件
- 创作输入组件
- 模式切换组件
- 参数面板组件
- 结果展示组件
- 历史记录组件
- 图片生成 API Route
- 视频生成 API Route

## 11. 非目标

本次设计不包含以下内容：

- 复杂账户体系
- 多人协作
- 长期持久化历史记录
- 后台管理系统
- 过多高级模型调参项

## 12. 结论

本方案将页面定义为一个单页图片与视频生成工作台，通过统一创作输入、模式切换、分栏工作台和会话历史记录，让用户在一个安静、克制、自然的界面中完成多模态创作。

技术上采用 Next.js App Router 与服务端 API Route 代理结构，保证密钥安全和后续接口演进的灵活性。

## 13. 增量调整（2026-03-26 晚）

根据后续确认，本页面在实现时还需要满足以下增量约束：

### 13.1 横向工作台布局

- 页面主工作区不再采用单纯纵向堆叠
- `参数与提交`、`当前结果`、`历史记录` 三个模块需要并列出现
- 主工作区采用横向滑动轨道，PC 端和移动端都支持左右切换
- 使用平滑滚动与 `scroll-snap` 保证切换顺滑，但不引入复杂重动画

### 13.2 移动端兼容

- 移动端卡片宽度应贴近视口，允许手势左右滑动
- 桌面端仍保持并列大卡片，但同样支持横向滚动体验
- 禁止出现小屏下卡片挤压变形或操作按钮过密的情况

### 13.3 历史记录增强

- 历史记录与当前结果区保持并列，而不是下沉到页面底部
- 每条历史记录需要显示资源链接
- 链接应可直接点击查看图片或视频
- 继续保留 `复用 prompt` 操作

### 13.4 视频参数收敛

- 视频生成功能固定为 8 秒
- 前端不再提供时长调节
- 服务端调用时按固定时长策略发起请求

### 13.5 输入区重设计

- 当前普通表单式文本框需要改为更精致的创作输入区
- 输入区应更接近编辑器观感，具备更强视觉层次
- 需要更好的聚焦态、留白和内层容器样式，避免生硬矩形输入框
